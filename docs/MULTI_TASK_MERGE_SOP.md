# 多任务并行开发与 Main 分支合并 SOP

## 1. 目的与适用范围

本 SOP 用于规范多个 Codex 任务并行修改同一项目时的任务登记、分支创建、开发、同步、验收和合并流程。

目标是：

- 保证 `main` 始终可以构建、运行，原则上可以随时发布；
- 减少任务之间的重复开发、文件冲突和依赖混乱；
- 确保每次合并都有明确范围、验证结果和回滚边界；
- 让新任务能快速了解项目当前状态并安全继续工作。

所有任务开始工作前，必须完整阅读：

1. `docs/MULTI_TASK_MERGE_SOP.md`
2. `docs/TASK_STATUS.md`

所有代码、分支、提交、测试和合并操作都必须在本真实 Git 仓库中完成，不得把 ChatGPT 项目上下文镜像当作代码仓库或跨任务共享目录。

## 2. 核心原则

1. 一个需求对应一个任务分支和一个独立工作区。
2. 功能任务不得自行直接修改或合并 `main`。
3. `main` 中的每个提交都必须可构建、可运行、可验证、可回滚。
4. 任务只有在“完成、可测、独立、最新、可回滚”时才允许合并。
5. 未完成但确需提前进入 `main` 的功能，必须向后兼容并由功能开关关闭。
6. 任务分支负责解决自身与最新 `main` 的冲突，不能把冲突留给集成任务。
7. 功能任务不得夹带与需求无关的重构、格式化或清理。
8. 集成任务一次只合并一个任务；每次合并后验证 `main`，再处理下一个任务。
9. 所有进行中和已合并任务都必须登记在 `docs/TASK_STATUS.md`。
10. `docs/TASK_STATUS.md` 只由集成任务维护，功能任务通过状态报告申请更新。

合并判断公式：

```text
完成 + 可测 + 独立 + 最新 + 可回滚 = 可以合并
```

## 3. 角色与责任

### 3.1 功能任务

功能任务负责：

- 阅读本 SOP 和最新状态表；
- 确认当前 `main` 基线、已有任务、依赖和潜在文件冲突；
- 明确需求范围、验收标准和预计修改区域；
- 从规定基准创建独立分支和工作区；
- 只实现本任务范围内的内容；
- 编写或更新必要测试；
- 同步最新 `main` 并自行解决冲突；
- 提交状态报告和完整交付报告；
- 未经明确授权，不得合并到 `main`；
- 不得直接修改中央状态表。

功能任务必须在以下时间向集成任务上报：

- 任务开始；
- 功能范围或主要修改区域变化；
- 新增或解除依赖；
- 被阻塞；
- 准备进入合并队列；
- 被退回后重新提交；
- 任务取消。

### 3.2 集成任务

集成任务是唯一统一管理合并的任务，负责：

- 维护 `docs/TASK_STATUS.md`；
- 登记任务范围、分支、依赖、主要文件和状态；
- 识别重复开发和潜在文件冲突；
- 确定合并顺序；
- 检查任务是否满足合并准入条件；
- 经用户明确授权后逐一合并；
- 每次合并后运行项目级验证；
- 记录准确的 `main` 提交 SHA；
- 合并失败时停止队列，修复或回滚后再继续；
- 通知其他待合并任务同步最新 `main`。

集成任务原则上不同时承担大规模功能开发，避免开发者和合并裁决者职责混杂。

### 3.3 中央任务状态表

`docs/TASK_STATUS.md` 是多任务协作状态的唯一中央索引，其必要性在于：

- 避免不同任务重复实现同一功能；
- 提前识别相同文件或模块的并行修改；
- 记录依赖关系和建议合并顺序；
- 区分“开发完成”“待合并”和“已进入 main”；
- 记录已合并功能及对应提交，便于追踪和回滚。

三类事实来源必须区分：

- Git 提交：代码变更的事实来源；
- `docs/TASK_STATUS.md`：任务状态、依赖和合并安排的事实来源；
- 功能任务交付报告：测试、风险和回滚方式的事实来源。

为避免中央表本身产生并行冲突，功能任务不得直接修改它，由集成任务统一更新。

## 4. 分支与提交规则

### 4.1 分支命名

```text
task/<简短需求名称>
fix/<简短问题名称>
refactor/<简短重构名称>
```

示例：

```text
task/coin-animation
task/score-system
fix/double-click
refactor/game-state
```

不得使用 `test`、`temp`、`new` 等含义不清的名称。

### 4.2 基准登记

任务启动时必须报告：

```text
任务 ID：
任务名称：
基准分支：
基准提交 SHA：
任务分支：
功能范围：
主要文件或模块：
依赖任务：
验收标准：
```

默认从最新 `main` 创建分支。只有存在明确依赖并得到集成安排时，才能基于其他任务分支。

### 4.3 提交规则

- 每个提交应有清晰、单一的目的；
- 不提交调试输出、缓存、临时文件、密钥或本地配置；
- 不使用 `update`、`fix`、`try again` 等无意义说明；
- 默认使用 squash 合并，使一个需求对应 `main` 中的一个提交；
- 推荐提交格式：

```text
feat: 添加硬币翻转动画
fix: 防止连续点击重复结算
refactor: 拆分游戏状态管理
test: 补充积分结算测试
docs: 更新多任务合并规范
```

## 5. 任务启动流程

每个任务开始修改前必须：

1. 阅读本 SOP；
2. 阅读最新 `docs/TASK_STATUS.md`；
3. 确认真正的 Git 仓库、当前分支和工作树状态；
4. 获取最新 `main` 并记录基准 SHA；
5. 明确验收标准；
6. 检查是否依赖未合并任务；
7. 检查预计修改区域是否与进行中任务重叠；
8. 向集成任务提交任务登记信息；
9. 确认集成任务已完成登记；
10. 创建独立分支和独立工作区后开始修改。

如果多个任务需要大量修改相同核心文件，应暂停并调整拆分。优先抽出公共接口或基础能力任务，合入 `main` 后再并行。

## 6. 开发过程要求

- 只修改完成本需求所必需的文件；
- 保持对现有功能的兼容；
- 公共接口变更必须说明影响范围；
- 数据、配置或数据库变更必须提供兼容和回滚方案；
- 新增或修改的核心逻辑必须补充测试；
- 需求外问题单独记录，不顺手扩大任务范围；
- 长周期任务定期同步最新 `main`；
- 范围、依赖、阻塞或主要修改区域变化时，立即上报集成任务。

### 6.1 未完成能力提前合入

只有同时满足以下条件才允许：

- 当前提交本身可构建、可运行；
- 不改变现有用户行为；
- 公共接口向后兼容；
- 未完成功能由 feature flag 或等效机制关闭；
- 有独立测试和明确回滚方式。

禁止把必须与另一个分支同时合并才能运行的半成品提交到 `main`。

## 7. 合并准入条件

### 7.1 需求完成

- 验收标准全部满足；
- 没有临时实现代替正式实现；
- 没有未说明的阻塞问题；
- 相关文档和配置已更新。

### 7.2 代码完整

- 可构建、可启动；
- 没有调试代码、假数据、临时文件或敏感信息；
- 没有无关重构和大范围格式化；
- 接口、配置和数据变更向后兼容；
- 实际改动范围与任务说明一致。

### 7.3 验证通过

- 新增和修改的测试通过；
- 相关模块测试通过；
- 项目构建和基础冒烟测试通过；
- 关键用户流程已验证；
- 已记录实际执行的测试及结果。

### 7.4 分支最新

- 已同步最新 `main`；
- 冲突已在任务分支解决；
- 同步后重新运行必要测试；
- 不存在未归属或未提交修改。

### 7.5 依赖明确

- 所有前置依赖已进入 `main`；或者
- 任务完全独立；或者
- 未完成功能被安全地置于功能开关之后。

### 7.6 可以回滚

- 任务可以作为一个独立变更撤销；
- 回滚不依赖大量手工恢复；
- 数据和配置变更有回滚或向后兼容方案。

## 8. 禁止合并的情况

出现以下任一情况，不得合并：

- 主体代码写完但尚未测试；
- 必须同时合并另一个未完成分支才能运行；
- 仍使用临时接口、假数据或硬编码方案；
- 合并后会让现有功能暂时不可用；
- 尚未同步最新 `main`；
- 存在未解决冲突；
- 依赖关系或合并顺序不明确；
- 包含未授权的额外重构或格式化；
- 构建、测试或冒烟验证失败；
- 无法说明回滚方式；
- 交付报告不完整。

## 9. 合并顺序

默认顺序：

1. 向后兼容的公共接口、数据结构和基础设施；
2. 核心业务逻辑；
3. 相互独立的功能；
4. UI、动画、音效等表现层；
5. 重构、格式整理和清理；
6. 独立文档变更。

显式依赖优先于默认顺序。

多个任务修改相同文件时：

1. 优先合并依赖层级更低、范围更小或优先级更高的任务；
2. 合并并验证 `main`；
3. 后续任务同步最新 `main`；
4. 后续任务自行解决冲突并重新测试；
5. 重新提交合并检查。

禁止批量合入多个冲突分支后再统一修复。

## 10. 标准合并流程

### 10.1 功能任务交付

1. 完成需求和测试；
2. 同步最新 `main`；
3. 解决冲突；
4. 重新执行测试和冒烟验证；
5. 检查最终差异；
6. 提交完整交付报告；
7. 请求集成任务将状态更新为“待合并”；
8. 等待处理，不自行合并。

### 10.2 集成任务检查

1. 检查交付报告；
2. 核对任务范围和实际改动；
3. 确认前置依赖已合并；
4. 确认分支基于最新 `main`；
5. 确认测试结果；
6. 必要时补充独立验证；
7. 更新状态、依赖和建议合并顺序；
8. 将任务加入合并队列。

### 10.3 合并与验证

1. 一次只处理一个任务；
2. 默认 squash 合并；
3. 合并后立即验证 `main`；
4. 通过后记录功能摘要、来源分支、提交 SHA 和验证结果；
5. 将任务移至“已合并”并更新 `main` 基线；
6. 通知其他待合并任务同步最新 `main`；
7. 更新推荐下一合并任务；
8. 再处理下一个任务。

```text
任务完成
  ↓
同步最新 main 并解决冲突
  ↓
测试与人工验收
  ↓
提交交付报告
  ↓
集成任务更新 TASK_STATUS
  ↓
检查准入条件
  ↓
Squash 合并到 main
  ↓
验证 main
  ↓
记录提交并通知其他任务
```

## 11. 合并失败与回滚

如果合并后 `main` 构建、测试或关键流程失败：

1. 立即停止后续合并；
2. 判断能否在极短时间内安全修复；
3. 原因明确且风险低时，可创建独立修复提交；
4. 否则优先回滚刚合入的任务；
5. 将问题退回原任务分支；
6. 修复后同步最新 `main` 并重新验证；
7. 再次进入合并队列；
8. 更新状态表中的状态、原因和后续处理。

不得在已损坏的 `main` 上继续叠加其他任务。

## 12. 紧急修复

1. 从最新 `main` 创建 `fix/<问题名称>`；
2. 只修改解决问题所必需的内容；
3. 补充复现问题的测试；
4. 完成构建、相关测试和冒烟验证；
5. 由集成任务优先合并；
6. 合并后通知所有进行中任务同步最新 `main`。

紧急修复不免除测试、回滚和交付报告要求。

```text
- docs/TASK_STATUS.md

检查状态表中的现有任务、依赖和修改范围，并向集成任务提交登记信息。

如果存在文件冲突或前置依赖，请先报告，不要自行扩大修改范围。
```
